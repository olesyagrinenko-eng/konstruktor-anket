# Спецификация интеграции конструктора подрядчика -> SSI JSON

Документ описывает единый JSON-контракт для импорта в `SSIGen`.

Версия контракта: `1.0.0`  
Дата обновления: `2026-05-04`  
Совместимость со схемой: `docs/contractor-ssi-questionnaire.schema.json`

## 1. Файлы

- Схема валидации: `docs/contractor-ssi-questionnaire.schema.json`
- Сводный файл примеров: `docs/contractor-ssi-questionnaire.examples.json`
- Примеры всех типов вопросов: `docs/question-type-examples/`
- Сводная таблица полей (Excel): `docs/contractor-ssi-fields-by-type.xlsx` — листы «Поля» (тип, допустимые значения, R/O/C по типам вопроса), «Квота ячейка», «Обозначения».

## 2. Текущие поддерживаемые типы вопросов

Поддерживаемые типы в нашей генерилке (`SSIGen`):

- `SELECT` - одиночный/множественный выбор (`selectFormat="1"` или `"2"`).
- `NUMERIC` - числовой ввод.
- `OPEN-END` - открытый текстовый ответ.
- `TEXT` - информационный/инструкционный экран без ответа.
- `GRID` - матричный вопрос со строками и колонками; на этом типе также может быть построен координатный клик-тест.
- `GRID_HYBRID` - гибридный матричный вопрос.
- `QUOTA` - квотный вопрос.

Что такое `GRID_HYBRID`:

- это несколько открытых числовых или текстовых полей по строкам;
- опционально может добавляться исключающий код, например, "Затрудняюсь ответить";
- базово передается через `list` (строки), при необходимости с `list_column` (колонки).

## 3. Корневой формат JSON

Корневой JSON-файл анкеты — это массив вопросов:

- тип корня: `array`
- минимальное количество элементов: `1`
- каждый элемент массива: объект одного вопроса (контракт вопроса описан ниже)

Минимальный каркас файла:

```json
[
  {
    "name": "Q1",
    "type": "TEXT",
    "header2": "Добро пожаловать в опрос"
  },
  {
    "name": "Q2",
    "type": "SELECT",
    "selectFormat": "1",
    "header2": "Ваш пол",
    "list": ["Мужской", "Женский"]
  }
]
```

## 4. Описание полей JSON

### Базовые поля вопроса

- `name: string` - код вопроса. Строгий формат: от `1` до `24` символов, первый символ — латинская буква, далее только латинские буквы и цифры (`^[A-Za-z][A-Za-z0-9]{0,23}$`). Значения, не соответствующие этому формату, считаются невалидными и отклоняются на этапе валидации.
- `type: string` - тип вопроса: `SELECT`, `NUMERIC`, `OPEN-END`, `TEXT`, `GRID`, `GRID_HYBRID`, `QUOTA`.
- `selectFormat: string | string[] | null` - формат выбора:
  - `"1"` - одиночный выбор,
  - `"2"` - множественный выбор,
  - `"NUMERIC"` / `"OPEN-END"` - тип строк в GRID-HYBRID вопросах,
  - массив (`string[]`) используется для построчной настройки `GRID_HYBRID`, например `["NUMERIC","2"]` или `["OPEN-END","2"]`,
  - `null` - если поле не используется.
- `header2: string | null` - основной заголовок/формулировка вопроса; допускается `null` (если строка — непустая, см. схему).
- `question: string | null` - дополнительная инструкция для респондента.
- `quota: object | null` - используется только при `type="QUOTA"`; для `QUOTA` обязателен и не может быть `null`.

### Поля списков

- `list: string[] | null` - основной список вариантов (для `SELECT`) или список строк (для `GRID`/`GRID_HYBRID`).
- `list_column: string[] | null` - список колонок (для `GRID` обязателен, кроме `click_coord`; для `GRID_HYBRID` опционален).
- `logo: boolean | null` - признак логотипов в обычном списке.
- `logoRows: boolean | null` - признак логотипов по строкам.
- `logoColumns: boolean | null` - признак логотипов по колонкам.

### Поля поведения списков

- `isRandomize: boolean | null` - ротация вариантов `list`.
- `isRandomize_row: boolean | null` - ротация строк.
- `isRandomize_column: boolean | null` - ротация колонок.
- `reverse: boolean | null` - реверс основного списка/строк.
- `reverse_column: boolean | null` - реверс колонок.
- `isExclusive: string` - индексы эксклюзивных вариантов через запятую, например `"9"` или `"8,9"`. Индексация вариантов фиксируется с `1`.
- `other: string` - индексы открытых (other/open) вариантов через запятую, например `"8"`. Индексация вариантов фиксируется с `1` (не с `0`).
- Для `GRID` и `GRID_HYBRID` индексы в `isExclusive` и `other` относятся к элементам `list` (строкам), а не к `list_column` (колонкам).
- Пример: если в `GRID_HYBRID` в `list` последняя строка — "Затрудняюсь ответить", то при длине `list = 11` исключающий код задается как `isExclusive="11"`.

### Валидация и ограничения ответа

- `isRequired: boolean | null` - обязательность ответа; для `type="TEXT"` по схеме допустимо, для инструкционных экранов обычно не передают.
- `minValue: number | number[] | null` - минимум (одно значение на вопрос или массив значений по строкам).
- `maxValue: number | number[] | null` - максимум (одно значение на вопрос или массив значений по строкам).
- `timeout: number | null` - таймаут перехода на следующий вопрос в секундах.
- Для `SELECT` поля `minValue` и `maxValue` интерпретируются как минимально и максимально допустимое количество выбранных вариантов.
- Для `SELECT` с `selectFormat="1"` значение `maxValue` обычно равно `1` (или `null`, если ограничение задается логикой одиночного выбора по умолчанию).
- Для `SELECT` с `selectFormat="2"` значения `minValue`/`maxValue` задают диапазон количества ответов (например, "выберите от 1 до 3").

### Правила для `GRID_HYBRID` (длины массивов и форматы)

- Базовое правило: если `selectFormat` передается как построчный массив, его длина должна совпадать с длиной `list`.
- Для `minValue` и `maxValue`: если передается массив, его длина должна совпадать с длиной `list` (по одному ограничению на строку).
- Если `minValue`/`maxValue` переданы как одно число, это общее ограничение для всех строк `list`.
- Допускается сокращенный формат из двух значений в `selectFormat`, например `["NUMERIC","2"]` или `["OPEN-END","2"]`, когда идет блок из нескольких подряд пустых вариантов ответа и в конце добавлен исключающий код (например, "Затрудняюсь ответить").
- В сокращенном формате `["NUMERIC","2"]`/`["OPEN-END","2"]`: первое значение задает тип полей для блока строк, второе (`"2"`) задает множественный режим с исключающим кодом.

### Поля `quota` (для `type="QUOTA"`) — базовый формат для клиента

Договоренность для интеграции: передаем только базовый формат квоты:

- `name`
- `type = "QUOTA"`
- `quota.cells: object[]` - непустой массив ячеек квоты
- `quota.cells[]` со следующими полями (все три **обязательны**):
  - `quota.cells[].cellLimit: integer` — лимит по ячейке (целое число)
  - `quota.cells[].condition: string` — условие попадания в ячейку; непустая строка (как и `label`, в схеме `minLength ≥ 1`)
  - `quota.cells[].label: string` — лейбл ячейки, отображаемый на квотной странице

Поля ячейки в SSI (`name`, числовой `value`, `comment` в терминах SSI, `alwaysQualify`) в JSON подрядчика **не передаются** — задаются при импорте; текст лейбла подрядчика попадает в описание ячейки в SSI.

Синтаксис условий для `quota.cells[].condition`:

- сравнения: `=`, `<=`, `>=`, `<>`
- логические операторы: `And`, `Or`, `Not()`
- пример: `QGender=1 And QAge>=18 And QAge<=34`

Пример квотного вопроса см. в `docs/question-type-examples/QUOTA.json`.

Runtime-валидация условий квоты:

- все переменные из `condition` должны существовать в анкете;
- все переменные из `condition` должны находиться выше (раньше) квотного вопроса в JSON.

### Поля `meta`

- `meta.widgetType: string` - спец-виджет: `scale` — шкальный вопрос, `click_coord` — координатный клик-тест.
- `meta.minScaleAnchor: string` - текстовый лейбл левого края шкалы, например "Совсем не нравится" (только для `widgetType="scale"`).
- `meta.maxScaleAnchor: string` - текстовый лейбл правого края шкалы, например "Очень нравится" (только для `widgetType="scale"`).
- `meta.mediaUrl: string` - ссылка на медиа-стимул (обязательна для `widgetType="click_coord"`).
- `meta.maxPoints: integer` - максимум кликов для `click_coord`.
- `meta.stimulusType: string` - тип стимула (`video`, `layout`, `scenario`, `concept`, `packaging`).
- `meta.stimulusIndex: integer` - номер стимула.

Если в `meta` задана ссылка `mediaUrl` (медиа-стимул / картинка), поля **`stimulusType`** и **`stimulusIndex`** обязательны (в т.ч. для `widgetType="click_coord"`, где `mediaUrl` уже обязателен). Без `mediaUrl` эти поля по схеме не требуются.

## 5. Обязательный минимум

Для каждого вопроса обязательны:

- `name` - код вопроса
- `type` - тип вопроса

Дополнительно по типам:

- для `SELECT` и `GRID_HYBRID` обязателен `list`
- для `GRID` обязательны `list` и `list_column`, **кроме** `meta.widgetType="click_coord"` — тогда оба поля могут отсутствовать (в сводной таблице Excel для столбца `GRID` у `list` и `list_column` стоит **C** — условная обязательность)
- для `QUOTA` обязателен объект `quota` с непустым массивом `quota.cells`

Дополнительно для координатного клик-теста (`meta.widgetType="click_coord"`):

- `type="GRID"`
- `meta.widgetType`
- `meta.mediaUrl`
- `meta.maxPoints`
- `meta.stimulusType` и `meta.stimulusIndex` (обязательны вместе с `mediaUrl`, см. выше)

## 6. Маппинг UI конструктора -> JSON

### Маппинг типов из UI конструктора

- `Открытый` -> `type="OPEN-END"`
- `Число` -> `type="NUMERIC"`
- `Один из списка` -> `type="SELECT"`, `selectFormat="1"`
- `Несколько из списка` -> `type="SELECT"`, `selectFormat="2"`
- `Мультивыбор (список в ТЗ)` -> `type="SELECT"`, `selectFormat="2"`
- `Шкала 1–9` -> `type="SELECT"`, `selectFormat="1"`, `list=["1","2","3","4","5","6","7","8","9"]`, `meta.widgetType="scale"`
- `Шкала 1–5` -> `type="SELECT"`, `selectFormat="1"`, `list=["1","2","3","4","5"]`, `meta.widgetType="scale"`
- `Инструкция` -> `type="TEXT"`
- `Клик-тест` -> `meta.widgetType="click_coord"`, `meta.mediaUrl`, `meta.maxPoints`, `meta.stimulusType`, `meta.stimulusIndex`

### Маппинг типов стимулов из шага "Материалы"

- `Ролики (видео)` -> `meta.stimulusType="video"`
- `Макеты (статика)` -> `meta.stimulusType="layout"`
- `Сценарии (текст/сториборд)` -> `meta.stimulusType="scenario"`
- `Концепции (сценарий + КС + визуал)` -> `meta.stimulusType="concept"`
- `Упаковки` -> `meta.stimulusType="packaging"`

Правила индексации стимула:

- порядковый номер стимула внутри типа -> `meta.stimulusIndex` (начиная с `1`)

### Маппинг полей вопроса UI

- `ID` -> `name`
- `Формулировка` -> `header2`
- `Тип` -> `type` + `selectFormat`
- `Варианты (через ;)` -> `list`
- `Шкала (якоря)` -> `meta.minScaleAnchor`, `meta.maxScaleAnchor` (только для `meta.widgetType="scale"`)
- медиа в карточке стимула -> `meta.mediaUrl`, `meta.stimulusType`, `meta.stimulusIndex`

## 7. Валидация JSON (рекомендуемый пример)

